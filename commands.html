Базові команди

Створення нового репозиторію

Як ми відзначили раніше, git зберігає свої файли та історію прямо в папці проекту. Щоб створити новий репозиторій, нам потрібно відкрити термінал, зайти в папку нашого проекту і виконати команду init. Це включить додаток в цій конкретній папці і створить приховану директорію .git, де буде зберігатися історія сховища та налаштування.

C:\Users\Desktop\htmlB\University>git init
Initialized empty Git repository in C:/Users/Margarita/Desktop/htmlB/University/.git/

Це означає, що наш репозиторій був успішно створений, але поки що порожній.

Визначення стану

status - це ще одна найважливіша команда, яка показує інформацію про поточний стан сховища: чи є актуальною інформація на ньому, чи немає чогось нового, що змінилося, і так далі.

git status
On branch master
Changes not staged for commit:
(use "git add ..." to update what will be committed)
(use "git checkout -- ..." to discard changes in working directory)
modified: css/styles.min.css
modified: css/styles.min.css.map
modified: index.html
modified: js/main.js
modified: quiz.html
modified: scss/modules/_commands-main.scss
modified: scss/modules/_quiz.scss
modified: setup.html
no changes added to commit (use "git add" and/or "git commit -a")

Повідомлення говорить про те, що деякі файли не відстежуються. Це означає, що файли новиі і система ще не знає, чи потрібно стежити за змінами у файлах або його можна просто ігнорувати. Для того, щоб почати відстежувати новий файл, потрібно його спеціальним чином оголосити.

Підготовка файлів

У git є концепція області підготовлених файлів. Можна уявити її як полотно, на яке наносять зміни, які потрібні в комміті. Спершу він порожній, але потім ми додаємо на нього файли (або частини файлів, або навіть поодинокі рядки) командою add і, нарешті, коммітимо все потрібне в репозиторій (створюємо зліпок потрібного нам стану) командою commit. Додамо один з файлів:

git add base.html

Якщо нам потрібно додати всі, що знаходиться в директорії, ми можемо використовувати

git add .

Де "." означає що необхідно додати все.

Комміт(фіксація змін)

Комміт є станом сховища в певний момент часу. Це схоже на снапшот, до якого ми можемо повернутися і побачити стан об'єктів на певний момент часу. Щоб зафіксувати зміни, нам потрібна хоча б одна зміна в області підготовки (ми тільки що створили її за допомогою git add), після якого ми можемо робити комміт:

git commit -m "add base.html"

Ця команда створить новий комміт з усіма змінами з галузі підготовки (додавання файлу base.html). Ключ -m - це створенbq користувачем опис всіх змін, включених в комміт. Вважається доброю практикою робити комміти часто і завжди писати змістовні коментарі..

Віддалені репозиторії

Підключення до віддаленого репозиторію

Щоб завантажити що-небудь в віддалений репозиторій, спочатку потрібно до нього підключитися. Для того щоб до нього підключитись потрібно стоврити аккаун та репозиторій на GitHub або ыншому сервері. Реєстрація та установка може зайняти час, але всі подібні сервіси надають хорошу документацію. Щоб зв'язати наш локальний репозиторій з репозиторієм на GitHub, виконаємо наступну команду в терміналі. URL сховища буде відрізнятись від вказаного

git remote add origin https://github.com/margarita666/E-course.git

Проект може мати кілька віддалених репозиторіїв одночасно. Щоб їх розрізняти, ми дамо їм різні імена. Зазвичай головний репозиторій називається origin.

Відправка змін на сервер

Зараз саме час переслати наш локальний комміт на сервер. Цей процес відбувається кожного разу, коли ми хочемо оновити дані в віддаленому репозиторії. Команда, призначена для цього - push. Вона приймає два параметри: ім'я віддаленого сховища (ми назвали наш origin) і гілку, в яку необхідно внести зміни (master - це гілка за замовчуванням для всіх репозиторіїв).

git push origin master
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 4 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (3/3), 309 bytes | 103.00 KiB/s, done.
Total 3 (delta 2), reused 0 (delta 0)
remote: Resolving deltas: 100% (2/2), completed with 2 local objects.
To https://github.com/margarita666/E-course.git
9d67c6e..87a5a94 master -> master

Залежно від сервісу, який ви використовуєте, вам може знадобитися аутентифікуватися, щоб зміни вирушили. Якщо все зроблено правильно, то коли ви подивіться в віддалений репозиторій за допомогою браузера, ви побачите додані раніше файли

Клонування репозиторію

Зараз інші користувачі GitHub можуть переглядати ваше сховище. Вони можуть завантажити з нього дані і отримати повністю працездатну копію вашого проекту за допомогою команди clone. Також ви самі можете стористатись цією командою працюючи, наприклад, з різних девайсів.

git clone https://github.com/margarita666/E-course.git

Новий локальний репозиторій створюється автоматично з GitHub в якості віддаленого сховища.

git add .

Де "." означає що необхідно додати все.

Запит змін з серверу

Якщо ви зробили зміни в вашому віддаленому репозиторії, ви можете завантажити зміни на інший девайс за допомогою команди pull.

git pull origin master
From https://github.com/margarita666/E-course.git
* branch master -> FETCH_HEAD
Already up-to-date.

Так як нових коммітів з тих пір, як ми склонували собі проект, не було, то ніяких змін доступних для скачування немає.

Розгалудження

Створення нової гілки

Основна гілка в кожному сховищі називається master. Щоб створити ще одну гілку, використовуємо команду branch -name

git branch amazing_new_feature

Це створить нову гілку, поки що точну копію гілки master.

Зміна гілки

Зараз, якщо ми запустимо branch, ми побачимо дві доступні опції:

git branch
amazing_new_feature
* master

master - це активна гілка, вона позначена зірочкою. Але ми хочемо працювати з новою гілкою, так що нам знадобиться переключитися на іншу. Для цього скористаємося командою checkout, вона приймає один параметр - ім'я гілки, на яку необхідно переключитися.

git checkout amazing_new_feature

Злиття гілок

На нашій новій гілці буде ще одинм файл під назвою feature.html. Ми створимо його, додамо і додамо в комміт:

git add feature.html
git commit -m "New feature complete."

Зміни завершені, тепер ми можемо перейти назад на гілку master.

git checkout master

Тепер, якщо ми відкриємо наш проект в файловому менеджері, ми не побачимо файлу feature.html, тому що ми переключилися назад на гілку master, в якій такого файлу не існує. Щоб він з'явився, потрібно скористатися merge для об'єднання гілок (застосування змін з гілки amazing_new_feature до основної версії проекту).

git merge amazing_new_feature

Тепер гілка master актуальна. Гілка amazing_new_feature більше не потрібна, і її можна видалити.

git branch -d awesome_new_feature

Додатково

Відстеження змін, зроблених в комміті

У кожного комміта є свій унікальний ідентифікатор у вигляді рядка цифр і букв. Щоб переглянути список всіх коммітів і їх ідентифікаторів, можна використовувати команду log:

git log
commit 87a5a941c275f9cd8897c96885c0bdf631e05734 (HEAD -> master, origin/master)
Author: Margarita
Date: Thu Nov 19 22:11:28 2020 +0200
add base.html
commit 9d67c6e358354c0da68ada54369f5bbab2d1a331
Author: margarita666_
Date: Thu Nov 19 19:53:47 2020 +0200
quiz done

Як ви можете помітити, ідентифікатори досить довгі, але для роботи з ними не обов'язково копіювати їх цілком - перших декількох символів буде цілком достатньо. Щоб подивитися, що нового з'явилося в комміті, ми можемо скористатися командою show [commit]

git show 9d67c6e
commit 9d67c6e358354c0da68ada54369f5bbab2d1a331
Author: margarita666_
Date: Thu Nov 19 19:53:47 2020 +0200
quiz done
diff --git a/css/styles.min.css b/css/styles.min.css
index 2464b9c..40c08f6 100644
--- a/css/styles.min.css
+++ b/css/styles.min.css

Щоб побачити різницю між двома коммітами, використовується команда diff (із зазначенням проміжку між коммітами):

git diff 87a5a941..9d67c6e
diff --git a/base.html b/base.html
index 765098e..b298912 100644
--- a/base.html
+++ b/base.html
@@ -25,7 +25,7 @@

Повернення файлу до попереднього стану

Гіт дозволяє повернути вибраний файл до стану на момент певного коммітов. Це робиться вже знайомої нам командою checkout, яку ми раніше використовували для переходу між гілками. Але вона також може бути використана для переходу між коммітами (це досить поширена ситуація для Гіта - використання однієї команди для різних, на перший погляд, слабо пов'язаних завдань). У наступному прикладі ми візьмемо файл index.html і відкотимо всі зміни, вчинені над ним до першого коммітов. Щоб зробити це, ми підставимо в команду ідентифікатор потрібного коммітов, а також шлях до файлу:

git checkout 09bd8cc1 index.html
